01 — Overview

What it does

One binary, three endpoints, no external dependencies at runtime beyond the standard library and gopsutil.

Node-level scraping

Reads CPU, memory, disk, load average, and process count from the host at a configurable interval.

Prometheus native

Exposes exposition format directly — hand-written, no client library — plus a JSON snapshot for humans.

Runs as a DaemonSet

Scheduled once per node by Kubernetes. Nodes join or leave — coverage follows automatically.

02 — Architecture

Collector → Exporter → HTTP

A background ticker scrapes the host on a fixed interval and stores the latest snapshot behind a mutex; handlers read that snapshot on request and never block on I/O.

03 — Endpoints

Three routes, nothing else

Route Format Purpose
/metrics text Prometheus exposition format — scraped by Prometheus or any compatible collector.
/snapshot json Same data, human-readable, useful for debugging without a Prometheus stack.
/health json Liveness check — hostname, environment, timestamp.
04 — Metrics

What's collected

Every metric carries hostname and env labels so values from different nodes stay distinguishable in a query.

Metric Type Description
agent_cpu_percent gauge Current CPU usage, percent.
agent_memory_used_percent gauge Memory in use, percent of total.
agent_memory_used_bytes gauge Memory in use, bytes.
agent_disk_used_percent gauge Disk usage per mount point, percent.
agent_disk_used_bytes gauge Disk usage per mount point, bytes.
agent_load_avg_1m / 5m / 15m gauge Standard Unix load averages.
agent_process_count gauge Number of running processes.
agent_uptime_seconds counter Time since the agent started.
05 — Deployment

One pod per node, by design

A Deployment schedules N replicas across available nodes — on a 10-node cluster, 3 replicas leave 7 nodes unmonitored. A DaemonSet has no such gap: node-level metrics need node-level presence.
